# Memory Tool -- Implementation Plan

## Original Prompt

> Store reusable, cached, snippets of information in a clear and easily readable manner.
>
> The intent is for an agent to quickly get up to speed on already known information without having to regather it.

### Architecture (from prompt)

- Store memory information in a SQLite file
- Use SQL queries to extract relevant information from the memory database
- Store the memory database in `.opencode/memory-db`
  - `.opencode/memory-db/AGENTS.md` - describes how to use the memory-db
  - `.opencode/memory-db/database.sqlite` - the sqlite file
- Database schema
  - id - unique id for the memory entry
  - Updated date - timestamp when the entry was last updated
  - Scopes - the relevant or applicable area(s) this entry pertains to
    - this could be a specific area of code such as: auth, api, design, structure
    - if this is an external reference and that external reference is versioned, the version should be stored. Multiple entries for the same external reference should be unique by version.
  - Content - the entry content - markdown format
  - Category - the category of the entry
    - user-requirement - the user has explicitly requested this be remembered; must always be included in context if working on the matching scopes
    - architecture - architectural format; not a hard requirement but worth checking if working in a similar area
    - design-pattern - pattern that could be reused or followed for consistency in the code base
    - reference - retrieved reference information, cached to avoid future costly lookups; an agent should check this prior to searching externally for similar information; these should be stored once confirmed correct by the agent
  - Archived - an archived entry is kept for tracking but should not be used in development of new changes, only as a historical reference if trying to debug _why_ something is a specific way
- This tool exposes tools for an agent to call
  - store-memory(scope, category, content) - store a new memory entry in the database, should internally make sure the memory doesn't already exist
  - find-memory(sql query) - takes any valid sqlite query to find a memory in the db
  - archive-memory(sql query or memory id) - mark a specific memory as archived

### Usage (from prompt)

- Agents can invoke these tools themselves while planning or building to store repeated information in a fast and concise manner.
- Retrieved external documentation can be cached here for future reference.
- Agents should check the memory first prior to reading code or fetching external documentation.
- If an agent has a piece of memory in its DB and it changes something that would make the memory outdated or irrelevant, the agent must update the memory.

## 1. File Layout

All new files live in `opencode/tools/` (symlinked to `~/.config/opencode/tools/`):

```
opencode/tools/
  memory.ts          # Tool definitions (store, find, archive)
  memory-db.ts       # Shared DB initialization, schema, helpers (not a tool -- no tool() export)
```

Static documentation template (generated on first DB init):

```
.opencode/memory-db/AGENTS.md           # Per-project (generated by tool)
~/.config/opencode/memory-db/AGENTS.md  # Global (generated by tool)
```

### Why `memory-db.ts` won't register as a tool

OpenCode registers exports that are `tool()` instances. A file that only exports plain functions and constants will not create any tools.

## 2. Database Schema

```sql
CREATE TABLE IF NOT EXISTS memories (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  updated_at  TEXT    NOT NULL DEFAULT (datetime('now')),
  scopes      TEXT    NOT NULL,  -- JSON array, e.g. '["auth","api"]'
  content     TEXT    NOT NULL,  -- Markdown format
  category    TEXT    NOT NULL CHECK(category IN (
                'user-requirement',
                'architecture',
                'design-pattern',
                'reference'
              )),
  archived    INTEGER NOT NULL DEFAULT 0  -- 0 = active, 1 = archived
);

CREATE INDEX IF NOT EXISTS idx_memories_category ON memories(category);
CREATE INDEX IF NOT EXISTS idx_memories_archived ON memories(archived);
```

Scopes are stored as a JSON array so agents can query with `json_each()`:

```sql
SELECT * FROM memories WHERE EXISTS (
  SELECT 1 FROM json_each(scopes) WHERE value = 'auth'
);
```

## 3. Dual-Database Strategy

| Database    | Path                                                     | Purpose                                               |
| ----------- | -------------------------------------------------------- | ----------------------------------------------------- |
| **Project** | `{context.worktree}/.opencode/memory-db/database.sqlite` | Project-specific patterns, architecture, requirements |
| **Global**  | `~/.config/opencode/memory-db/database.sqlite`           | Cross-project references, cached external docs        |

### Routing logic

- `store` and `archive` accept an optional `global` boolean (default `false`). When `true`, operations target the global DB.
- `find` accepts an optional `global` boolean. When `true`, queries the global DB. When `false` (default), queries the project DB.
- Both databases use the same schema.

### Initialization

On first access to either database:

1. Create the directory (`mkdirSync` with `recursive: true`)
2. Open/create the SQLite file
3. Run the `CREATE TABLE` and `CREATE INDEX` statements
4. Write the `AGENTS.md` file into the same directory (if not already present)

## 4. Tool Definitions

### `memory_store`

Named export `store` from `memory.ts`.

| Arg        | Type      | Required | Description                                                       |
| ---------- | --------- | -------- | ----------------------------------------------------------------- |
| `scopes`   | `string`  | Yes      | Comma-separated scope names (e.g. "auth,api")                     |
| `category` | `enum`    | Yes      | One of: user-requirement, architecture, design-pattern, reference |
| `content`  | `string`  | Yes      | Markdown content to store                                         |
| `global`   | `boolean` | No       | Store in global DB instead of project DB (default: false)         |

Logic:

1. Parse `scopes` string into a JSON array (split on comma, trim whitespace)
2. Check for duplicate: `SELECT id FROM memories WHERE content = ? AND scopes = ? AND category = ? AND archived = 0`
3. If duplicate exists, update `updated_at` and return the existing entry
4. Otherwise, `INSERT` and return the new entry with its id

Returns: JSON with `status`, `id`, `message`

### `memory_find`

Named export `find` from `memory.ts`.

| Arg      | Type      | Required | Description                                            |
| -------- | --------- | -------- | ------------------------------------------------------ |
| `query`  | `string`  | Yes      | Valid SQLite SELECT query against the `memories` table |
| `global` | `boolean` | No       | Query global DB instead of project DB (default: false) |

Logic:

1. Validate the query starts with `SELECT` (case-insensitive, trimmed) -- reject writes
2. Execute the query with `.all()`
3. Return results as JSON array

Returns: JSON with `status`, `results` array, `count`

### `memory_archive`

Named export `archive` from `memory.ts`.

| Arg            | Type      | Required | Description                                           |
| -------------- | --------- | -------- | ----------------------------------------------------- |
| `id`           | `number`  | No       | Specific memory ID to archive                         |
| `scope`        | `string`  | No       | Archive entries that include this scope               |
| `category`     | `enum`    | No       | Archive entries matching this category                |
| `contentMatch` | `string`  | No       | Archive entries whose content contains this substring |
| `global`       | `boolean` | No       | Target global DB (default: false)                     |

Logic:

1. Require at least one of `id`, `scope`, `category`, or `contentMatch`
2. Build parameterized WHERE clause from provided filters (all conditions AND'd together)
3. Execute `UPDATE memories SET archived = 1, updated_at = ... WHERE <conditions>` with bound parameters
4. Return count of archived entries

Returns: JSON with `status`, `archived_count`, `message`

### `memory_list_scopes`

Named export `list_scopes` from `memory.ts`.

| Arg      | Type      | Required | Description                                            |
| -------- | --------- | -------- | ------------------------------------------------------ |
| `global` | `boolean` | No       | Query global DB instead of project DB (default: false) |

Logic:

1. Query all distinct scope values from non-archived entries using `json_each()`
2. Return each scope with a count of how many entries it appears in, ordered by count descending

Returns: JSON with `status`, `results` (array of `{ scope, count }`), `count`

### `memory_list_categories`

Named export `list_categories` from `memory.ts`.

| Arg      | Type      | Required | Description                                            |
| -------- | --------- | -------- | ------------------------------------------------------ |
| `global` | `boolean` | No       | Query global DB instead of project DB (default: false) |

Logic:

1. Query all distinct categories from non-archived entries
2. Return each category with a count of entries, ordered by count descending

Returns: JSON with `status`, `results` (array of `{ category, count }`), `count`

## 5. AGENTS.md Template

Generated into each memory-db directory on first init. Describes:

- What the memory database is and how to use it
- The schema (table structure, categories, scopes format)
- Available tools and their arguments
- When to check memory (before external lookups, when planning)
- When to store memory (after confirming external docs, user requirements, discovered patterns)
- When to archive (when information becomes outdated)
- Example queries for common operations

## 6. Implementation Order

1. `memory-db.ts` -- Shared module: DB open/init, schema creation, AGENTS.md generation, path resolution
2. `memory.ts` -- Five named tool exports (`store`, `find`, `archive`, `list_scopes`, `list_categories`) using the shared module
3. Verify -- Manual testing by invoking the tools in an OpenCode session

## 7. Considerations

### Gitignore

The `.opencode/` directory is typically gitignored. The `AGENTS.md` file could optionally be committed if the team wants shared agent documentation.

### No dependency installation needed

Bun's built-in `bun:sqlite` module means zero external dependencies.

### Connection management

Each tool invocation opens the database, performs the operation, and closes it. No connection pooling needed.

### Safety on find

Validates queries start with `SELECT` to prevent mutations through the query interface.

### Safety on archive

Uses structured parameterized filters (`id`, `scope`, `category`, `contentMatch`) instead of raw SQL -- eliminates SQL injection.
